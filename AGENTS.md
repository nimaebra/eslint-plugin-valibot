# Repository Agent Guide

This file is the maintainer-oriented working guide for coding agents operating in this package.

## Scope

- Package: `eslint-plugin-valibot`
- Current package version: `1.0.0`
- Purpose: ESLint rules for safer and more maintainable Valibot usage.
- Runtime model: published as a Node ESM/CJS library built from TypeScript.
- Package manager: `pnpm`
- Supported Node versions: `^22.13.0 || >=24.0.0`
- Supported peers: `eslint@^9 || ^10`, `valibot@^1.0.0`

## Package Snapshot

- Entry source: `src/index.ts`
- Built outputs: `dist/index.mjs`, `dist/index.cjs`, `dist/index.d.ts`
- Package exports only the root entry and `./package.json`
- `sideEffects` is `false`
- Build tool: `tsup`
- Test runner: `vitest`
- Lint/type stack: ESLint 10, `typescript-eslint`, TypeScript 6
- Release management: Changesets plus GitHub Actions

## Public API Surface

`src/index.ts` re-exports the package's supported public API:

- Default export: the plugin object from `src/plugin.ts`
- Named exports: `configs`, `flatConfigs`
- Named rule exports for a manually maintained subset of rule modules
- Registry helpers: `ruleRegistry`, `rules`, `getRulesForConfig`, `getRuleNamesForConfig`
- Type exports from `src/types.ts`

Do not assume a rule exported from `src/rules/index.ts` is automatically re-exported from the package root. Root named exports must be added explicitly in `src/index.ts`.

`src/plugin.ts` builds the plugin shape from three pieces:

- `rules` from `src/rules/registry.ts`
- legacy configs from `src/configs/legacy`
- flat configs from `src/configs/flat`

The plugin metadata version is read from `package.json`.

## Architecture

### Core Control Plane

`src/rules/registry.ts` is the source of truth for most repo behavior.

It controls:

- which rules are exposed by the plugin
- which presets include each rule
- each preset severity level
- which rules are TypeScript-only
- the config docs generated by `scripts/config-docs.ts`
- config consistency checks in `scripts/check-configs.ts`
- integration test expectations in `tests/integration/examples.test.ts`

If a rule exists but is missing from the registry, the package is incomplete.

### Config Layers

- `src/configs/flat/*` builds flat config arrays
- `src/configs/legacy/*` builds legacy config objects
- both config systems derive their `rules` payload from `getRulesForConfig(...)`

This means preset membership should normally be changed in `src/rules/registry.ts`, not by hand-editing multiple config files.

### Shared Rule Utilities

Common helpers live in `src/utils`.

Important ones:

- `create-rule.ts`: wraps `ESLintUtils.RuleCreator` and standardizes docs URLs
- `collect-valibot-imports.ts`: finds Valibot imports for both ESM and CommonJS patterns
- `is-valibot-call.ts`: resolves whether a call expression maps to a Valibot API name
- `pipe-analysis.ts`: normalizes `pipe()` action analysis
- `schema-identifiers.ts`: recognizes schema bindings
- `type-aware.ts`: safely probes parser services and type information availability
- `issue-message-signatures.ts`: maps Valibot APIs to the issue-message argument position used by `require-issue-messages`

Prefer reusing these helpers over duplicating AST matching logic.

### Tests And Fixtures

- `tests/rules`: rule-level behavior tests
- `tests/configs`: flat/legacy config parity tests
- `tests/integration`: end-to-end tests against the built package and example apps
- `examples/flat`, `examples/legacy`, `examples/typescript`: public examples and integration fixtures

Integration tests lint the example projects against `dist/index.mjs`, not the source tree. If package behavior changes, build before running integration coverage.

### Docs And Generation

- `README.md` contains an auto-generated rules list maintained by `eslint-doc-generator`
- `docs/configs.md` is generated from the rule registry by `pnpm docs:build`
- `docs/rules/*.md` are per-rule docs and must exist for every registry entry

Rule docs contain auto-generated markers that must remain intact:

- `<!-- end auto-generated rule header -->`
- `<!-- end auto-generated rule options -->`

## Repo Invariants

Treat the following as hard requirements:

1. `src/rules/registry.ts` must stay aligned with implemented/exported rules.
2. `src/rules/index.ts` must export every supported rule that should be public.
3. Flat and legacy preset contents must stay in sync with the registry.
4. Every registry rule must have a matching doc file in `docs/rules`.
5. Changes to preset membership or severities usually require docs regeneration and integration fixture updates.
6. Integration tests use the built package, so `pnpm build` is required before meaningful `pnpm test:integration` runs.
7. The published package only ships `dist`, so source-only fixes are insufficient until the package builds cleanly.

## Implementation Conventions

### Rule Authoring

- Use `createRule(...)` from `src/utils/create-rule.ts`
- Keep rule metadata accurate because docs tooling depends on it
- Reuse shared Valibot import/call helpers instead of open-coding identifier matching
- Short-circuit early when a file has no Valibot imports
- Keep fixes and suggestions minimal and syntax-safe

### Type-Aware Rules

- Use helpers from `src/utils/type-aware.ts` when parser services may be unavailable
- Do not assume type information is configured in every lint environment
- `no-schema-as-type` is the current TypeScript-only rule and is marked with `typeScriptOnly: true` in the registry

### Source Style

- This is a TypeScript ESM codebase
- Prefer `import type` where the symbol is type-only
- Keep changes focused and avoid unrelated refactors
- Follow existing naming and visitor structure patterns in nearby rules
- Preserve generated markers in docs

### Examples And Fixtures

- Example projects are user documentation and test fixtures at the same time
- Preset changes can force updates in `examples/*/src/*invalid*` files because integration tests compare exact rule IDs
- The stylistic flat example is expected to emit warnings without failing the ESLint process, so tests inspect reported rule IDs even when the CLI exit status is `0`

## Validation Guidance

Use the narrowest command that can falsify the change first, then widen only as needed.

### Rule Logic Change

Recommended order:

1. `pnpm exec vitest run tests/rules/<rule-name>.test.ts --reporter=verbose`
2. `pnpm test`

Add these when applicable:

- `pnpm build && pnpm test:integration` if preset behavior or packaging could change
- `pnpm docs:build && pnpm docs:check` if metadata or docs changed

### Registry Or Preset Change

Run:

1. `pnpm test`
2. `pnpm configs:check`
3. `pnpm docs:build`
4. `pnpm docs:check`
5. `pnpm build`
6. `pnpm test:integration`

### Packaging Or Release Change

Run:

1. `pnpm check`
2. `pnpm smoke:pack -- --valibot-version 1.0.0`
3. `pnpm smoke:pack -- --valibot-version latest`
4. `pnpm pack:dry-run`

## Change Playbooks

### Adding A New Rule

1. Optionally scaffold with `pnpm create:rule <rule-name>`.
2. Implement the rule under `src/rules/<rule-name>.ts`.
3. Export it from `src/rules/index.ts`.
4. If the rule should be a root named export, add it to `src/index.ts` too.
5. Register it in `src/rules/registry.ts` with the correct preset severities.
6. Set `typeScriptOnly: true` if it depends on TS syntax/type information.
7. Add or complete tests in `tests/rules/<rule-name>.test.ts`.
8. Add or complete docs in `docs/rules/<rule-name>.md`.
9. Run focused tests.
10. Run `pnpm docs:build` and `pnpm docs:check`.
11. If preset membership changed, run `pnpm build` and `pnpm test:integration` and update example fixtures as needed.

### Changing Rule Metadata Or Preset Membership

Start in `src/rules/registry.ts`.

Then verify:

1. config parity
2. docs regeneration
3. integration fixtures
4. README rules table if applicable

### Changing Public Exports

Touchpoints usually include:

- `src/index.ts`
- `src/plugin.ts`
- `src/rules/index.ts`
- `README.md`
- example configs if user-facing usage changes
- integration tests if built-package behavior changes

### Changing Generated Docs Or Config Output

Touchpoints usually include:

- `scripts/config-docs.ts`
- `scripts/build-configs-doc.ts`
- `scripts/check-docs.ts`
- `docs/configs.md`
- `README.md`

If the README rules table does not refresh after adding or exporting a rule, build first and rerun `pnpm docs:build`.

## Agent Working Rules For This Repo

1. Start from the owning rule, registry entry, config builder, or validation script rather than broad repo exploration.
2. If behavior differs between presets, inspect `src/rules/registry.ts` before editing config files.
3. If tests fail only in integration, inspect the built-package path plus `examples/*` fixtures before changing rule logic.
4. Reuse utilities in `src/utils` instead of duplicating Valibot AST detection.
5. Keep generated content and its generators in sync.
6. Do not hand-edit `dist`.
7. Do not ship a user-facing change without matching tests and docs.

## High-Risk Areas

- registry drift from actual exports
- flat and legacy preset divergence
- missing rule docs for a registered rule
- example fixtures becoming stale after preset changes
- forgetting to build before integration or packaged smoke tests
- changing rule names or export names without updating docs and examples

## Good First Inspection Points

When deciding where to start, use this map:

- Rule behavior bug: `src/rules/<rule>.ts` and `tests/rules/<rule>.test.ts`
- Preset mismatch: `src/rules/registry.ts`, `src/configs/flat/*`, `src/configs/legacy/*`, `tests/configs/plugin-configs.test.ts`
- Missing docs/config sync: `scripts/check-docs.ts`, `scripts/config-docs.ts`, `docs/configs.md`
- Built package or example failure: `pnpm build`, `tests/integration/examples.test.ts`, `examples/*`
- Release/publish concern: `package.json`, `.changeset/config.json`, `.github/workflows/release.yml`
